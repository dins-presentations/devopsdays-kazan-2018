Андрей:

мерж реквест -> прогон юнит-тестов и статической аналитики на девелоперском стейдже -> сборка артефакта (контейнера) -> прогон компонентных тестов (с моками) на командном стэйдже -> формируется заявка в планировщике релизов -> на основе информации из нашего планировщика релизов прогоняем тестов на контракты, с правильными версиями других сервисов -> если все ОК, принимаем мерж реквест -> артефакт/контейнер становится доступным для тестирования отдельной подсистемы, например какой-то части нашей API платформы -> прогон тестов на подсистемном стэйдже, окружение также формируется на основе информации из планировщика -> если есть ошибки, проблемные артефакты отклоняются и тестирование проводится снова -> если все ОК, артефакт/контейнер становится доступным для системного CI -> проводим оставшиеся виды тестирования, общую приемку и если все ОК, версия становится доступной всем в том числе и для деплоя на продакшн Т.е. в каждый момент времени мы имеем снапшот рабочей подсистемы и системы, которые могут быть использованы в процессе разработки и последующей непрерывной поставки.
